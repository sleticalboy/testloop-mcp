# 项目定位

## 一句话定位

testloop-mcp 是面向 AI Coding Agent 的测试反馈与质量控制 MCP 层，不是与 Claude Code、Cursor、Codex 或通用 LLM 竞争的独立测试生成器。

## 为什么不是单纯测试生成器

主流 AI 编程工具已经可以直接读代码、写测试、运行命令并根据失败结果继续修改。仅仅提供“根据源码生成测试用例”的能力，很容易变成重复建设：

- 通用 Agent 已经能生成测试草稿。
- 语言生态已有成熟工具，例如 Go 的静态测试骨架生成工具。
- 纯模板生成容易产生低价值测试，例如只覆盖 happy path、参数使用零值或占位值、断言过于宽泛。
- 单次生成测试并不能保证代码质量，真正困难的是持续运行、解析失败、定位覆盖缺口并推动修复闭环。

因此，本项目的核心价值不应放在“替 AI 写测试”，而应放在“让 AI 更稳定、更可控地使用测试反馈”。

## 核心价值

### 统一测试工具层

不同 AI 编程工具可以通过同一组 MCP 工具运行测试、解析结果、分析覆盖率和获取生成上下文，避免每个 Agent 都临时解析日志、临时决定测试命令。

### 结构化测试反馈

把 `go test`、Jest/Vitest、pytest、Mocha 等框架的原始输出转换为稳定 JSON，包括：

- 测试状态
- 失败测试名
- 失败文件和行号
- 断言消息
- expected / received
- 原始输出摘要
- 覆盖率缺口

这些信息比整段日志更适合 AI Agent 消化，也更适合自动化流程复用。

### 质量控制与策略沉淀

项目应沉淀团队级测试策略，而不是依赖每次 prompt 临场发挥，例如：

- 不生成会立刻失败的零值业务测试。
- 无法推断有效输入时，测试应标记为 TODO 或 skip。
- 优先针对覆盖率缺口生成测试任务。
- 失败修复应基于结构化错误和源码上下文，而不是盲改业务代码。
- 生成测试要区分真实行为验证和纯 mock 噪声。

### 覆盖率驱动的测试计划

长期方向不是“整文件泛化生成测试”，而是根据覆盖率缺口输出可执行测试计划：

- 目标函数或方法
- 未覆盖分支
- 建议输入形状
- 需要构造的依赖或 fixture
- 置信度
- 推荐执行命令

### 跨 Agent 可复用

testloop-mcp 不绑定某个 IDE 或某个模型。Claude Code、Cursor、Codex 或其他支持 MCP 的工具都可以复用同一套测试反馈能力。

## 非目标

以下方向不应成为项目主线：

- 为每门语言手写完整测试生成器。
- 追求替代通用 LLM 的语义测试生成能力。
- 只生成看起来有覆盖率但断言价值很低的测试。
- 把项目做成某个框架的单点 CLI 工具。

## 产品原则

1. 反馈闭环优先于生成数量。
2. 结构化结果优先于原始日志。
3. 覆盖率缺口优先于整文件扫描。
4. 可解释测试计划优先于黑盒生成。
5. 成熟生态工具优先集成，本地生成器作为回退。

## 后续路线约束

后续功能优先级应按以下顺序判断：

1. 是否能让 AI Agent 更准确地理解测试失败。
2. 是否能减少低价值或误导性测试。
3. 是否能把覆盖率缺口转化为具体修复或测试任务。
4. 是否能跨语言、跨 Agent 复用。
5. 是否避免把项目拖回“模板测试生成器”的方向。
